# design.md記載ガイド

## design.mdの位置づけ

AIと人間の共通理解を作るための「実行可能な設計ドキュメント」。
実装前に合意形成の場として機能し、AIへの指示の基盤となる。

## 書くべき内容

### 1. 概要（Overview）

- 何を作るのか
- 誰のためか
- どんな価値を提供するか

### 2. アーキテクチャ設計

- mermaidを使用しアーキテクチャ概要を記載
- コンポーネント構成・データの流れ・主要な依存関係を図示

### 3. 機能一覧

- 実装する機能をテーブルで列挙（機能ID・機能名・優先度・概要）

### 4. 画面・API設計

- 画面遷移図（フロントエンドがある場合、mermaid推奨）
- APIエンドポイント一覧（メソッド・パス・概要）

### 5. データモデル

- 主要エンティティとその関係をER図で図示（mermaid推奨）

### 6. 非機能要件

- パフォーマンス・セキュリティ・可用性などを数値で明示

### 7. 制約・前提条件

- スコープ外の機能
- 依存するシステム・サービス
- 技術的制約

## 注意事項

- [ ] 概要が3行以内にまとまっているか
- [ ] アーキテクチャ図（mermaid）が含まれているか
- [ ] 機能一覧が網羅されているか
- [ ] API設計が明確か（エンドポイント・メソッド・概要）
- [ ] データモデルが図示されているか
- [ ] 非機能要件が数値で明示されているか
- [ ] 制約・スコープ外が明記されているか
- [ ] 曖昧な表現がないか（「できれば」「なるべく」「適切な」等）
