# design.mdの書き方

## design.mdの位置づけ

AIと人間の共通理解を作るための「実行可能な設計ドキュメント」。
実装前に合意形成の場として機能し、AIへの指示の基盤となる。

## 保存先

`docs/design.md`

---

## 記載構成

### 1. 概要（Overview）

- 何を作るのか
- 誰のためか
- どんな価値を提供するか

**例：**
```markdown
## 概要

ECサイトの商品検索機能を構築する。
購買意欲のあるユーザーが素早く目的の商品に辿り着けるよう、
キーワード検索・絞り込みフィルター・並び替えを提供する。
```

---

### 2. アーキテクチャ設計

mermaidを使用してアーキテクチャ概要を図示する。

**記載すること：**
- コンポーネント構成（フロントエンド / バックエンド / DB / 外部サービス）
- データの流れ
- 主要な依存関係

**例：**
```markdown
## アーキテクチャ

\`\`\`mermaid
graph TD
  Client["ブラウザ / モバイル"]
  API["バックエンドAPI (FastAPI)"]
  DB["PostgreSQL"]
  Cache["Redis"]

  Client -->|HTTP/JSON| API
  API --> DB
  API --> Cache
\`\`\`
```

---

### 3. 機能一覧

実装する機能をテーブルで列挙する。

| 機能ID | 機能名 | 優先度 | 概要 |
|--------|--------|--------|------|
| F-001  | ○○機能 | 高     | 概要 |

---

### 4. 画面・API設計

#### 画面設計（フロントエンドがある場合）
- 画面一覧とそれぞれの役割
- 画面遷移図（mermaid推奨）

#### API設計（バックエンドがある場合）
- エンドポイント一覧

| メソッド | パス | 概要 |
|----------|------|------|
| GET      | /api/items | 商品一覧取得 |

---

### 5. データモデル

主要エンティティとその関係をER図で示す（mermaid推奨）。

```markdown
\`\`\`mermaid
erDiagram
  USER {
    int id PK
    string email
  }
  ORDER {
    int id PK
    int user_id FK
  }
  USER ||--o{ ORDER : "places"
\`\`\`
```

---

### 6. 非機能要件

| 項目 | 要件 |
|------|------|
| パフォーマンス | APIレスポンス 200ms以内（p95） |
| セキュリティ | OWASP Top10準拠 |
| 可用性 | 99.9%以上 |

---

### 7. 制約・前提条件

- スコープ外の機能
- 依存するシステム・サービス
- 技術的制約

## チェックリスト

- [ ] 機能一覧が網羅されているか
- [ ] API設計が明確か（エンドポイント・メソッド・概要）
- [ ] データモデルが図示されているか
- [ ] 非機能要件が数値で明示されているか
