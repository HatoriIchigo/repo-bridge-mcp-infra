## コミットメッセージ規則

### タイトル書式

```
<種別>: <要約>
```

- 1行目はタイトルのみ（50文字以内推奨）
- 種別と要約の間はコロン+半角スペース
- 要約は体言止め（「〜の修正」「〜の追加」）

### コミット種別

| 種別 | 用途 |
|------|------|
| `add` | 新規ファイル・機能の追加 |
| `update` | 既存機能の修正・改善（バグ以外） |
| `fix` | バグ修正 |
| `remove` | ファイル・コードの削除 |
| `refactor` | 外部仕様を変えないリファクタリング |
| `docs` | ドキュメントのみの変更 |
| `chore` | ビルド設定・依存更新など |
| `test` | テストの追加・修正 |

### 本文（Subject）

タイトルと本文は1行空けて区切る。

**書く内容**

- なぜその変更が必要だったか（理由・背景）
- 何をどう変えたか（概要）
- 影響範囲（あれば）

**書かない内容**

- テストがPASSしたかどうか
- 変更ファイルの列挙（`git diff` で確認できる情報）
- 「〜を修正しました」などの敬語

### コミットメッセージ例

#### バグ修正

```
fix: ログイン時にセッションエラーが発生する問題を修正

原因：
- セッションの有効期限チェックが漏れていた

対応：
- トークン検証ロジックを追加
- リフレッシュトークンの再取得処理を実装

影響範囲: 認証モジュール全体
```

#### 新規機能追加

```
add: CSV一括エクスポート機能の追加

背景：
- 管理者から大量データのダウンロード要望があった

対応：
- /api/export エンドポイントを追加
- ストリーミングレスポンスで大容量データに対応
```

#### リファクタリング

```
refactor: 認証サービスのメソッド分割

外部仕様に変更なし。
単一責任原則に従いUserServiceからAuthServiceを分離。
```

#### ドキュメント

```
docs: APIリファレンスにレート制限の説明を追加
```

### 注意事項

- [ ] タイトルが「種別: 要約」形式になっているか
- [ ] 種別が上記テーブルのいずれかと一致しているか
- [ ] タイトルが50文字以内か
- [ ] 本文がある場合、タイトルと1行空けているか
- [ ] 本文に「なぜ」が書かれているか
- [ ] 敬語・丁寧語を使っていないか
- [ ] `git diff` で分かる情報（変更ファイル一覧など）を重複記載していないか
